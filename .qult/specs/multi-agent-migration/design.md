# Design: multi-agent-migration

## Architecture

### 概観

移行後の qult は Node.js 20+ で動作する単一の npm パッケージ `qult` として配布される。現在の Claude Code Plugin アーキテクチャ（`plugin/` ディレクトリ、Bun ランタイム、`~/.claude/plugins/` 配布）を廃棄し、CLI ツール (`npx qult`) + MCP server (`npx qult mcp`) の二本柱に置き換える。

**主要な責務分離:**

```
ユーザー (npx qult init/update/check/add-agent/mcp)
    │
    ├─ CLI層 (src/cli/)
    │     コマンドパース → Integration Registry 呼び出し → 出力
    │
    ├─ Integration Registry (src/integrations/)
    │     自動検出 → テンプレート変換 → ファイル生成 → MCP 設定登録
    │
    ├─ Templates (src/templates/)
    │     agent-neutral テンプレート → {{VAR}} 置換 → 各ツール形式へ変換
    │
    ├─ State (src/state/)
    │     .qult/state/*.json の読み書き (atomic rename)
    │
    ├─ Detector (src/detector/)
    │     Tier 1 の 5 種類の検出器
    │
    └─ MCP Server (src/mcp/)
          stdio JSON-RPC → State/Spec 操作ツール群
```

### 依存関係図

```
src/cli/       → src/integrations/ → src/templates/
                                   → src/state/
               → src/detector/     → src/state/
               → src/state/

src/mcp/       → src/state/
               → src/detector/

src/integrations/ → src/templates/
                  → src/state/  (config 読み込みのみ)

src/state/     → node:fs (のみ; 外部依存ゼロ)
src/detector/  → node:child_process, node:fs, fetch
src/templates/ → (依存なし)
```

外部 npm 依存は開発時のみ (`tsup`, `typescript`, `vitest`, `@biomejs/biome`)。`dependencies` は空を維持する。

## Data Model

### `.qult/` ディレクトリ構造（移行後）

```
.qult/
├── config.json              # committed: プロジェクト設定
├── specs/                   # committed: spec markdown
│   ├── <spec-name>/
│   │   ├── requirements.md
│   │   ├── design.md
│   │   ├── tasks.md
│   │   └── waves/wave-NN.md
│   └── archive/
└── state/                   # gitignored: ephemeral
    ├── current.json
    ├── pending-fixes.json
    ├── stage-scores.json
    └── gates.json
```

### `.qult/config.json` スキーマ（拡張版）

既存の `QultConfig` 型に `integrations` セクションと `templates` セクションを追加する。

```typescript
interface QultConfig {
  // 既存フィールド（現 src/config.ts と同一）
  review: { score_threshold: number; max_iterations: number; ... };
  spec_eval: { ... };
  gates: { ... };
  security: { require_semgrep: boolean };

  // 新規追加
  integrations: {
    /** 有効な integration キー一覧。init/update が書き込む。 */
    enabled: IntegrationKey[];
    /** カスタム integration 定義。組み込みリストに追加される。 */
    custom?: CustomIntegrationDef[];
  };
  templates: {
    /** 組み込みテンプレートより優先するカスタムテンプレートディレクトリ（絶対/相対）。 */
    dir?: string;
  };
}

type IntegrationKey = "claude" | "codex" | "cursor" | "gemini" | string;

interface CustomIntegrationDef {
  key: string;
  displayName: string;
  /** MCP 設定ファイルのプロジェクト相対パス */
  mcpConfigPath: string;
  /** MCP サーバー設定の JSON スニペット（文字列化）*/
  mcpConfigSnippet: string;
}
```

### State ファイルスキーマ（既存と同一）

### AGENTS.md スキーマ

agentsmd.io 標準（`<!-- @generated by qult -->` マーカーブロック）を採用する。

```markdown
# AGENTS.md

<!-- ユーザー既存コンテンツがあればここに保持 -->

<!-- @generated by qult vX.Y.Z — DO NOT EDIT THIS SECTION -->
## qult Workflow

This project uses [qult](https://github.com/hir4ta/qult) for Spec-Driven Development (SDD).

### MCP Tools (via qult MCP server)
- `qult__get_project_status` — current spec phase, wave, test/review state
- `qult__get_active_spec` — active spec details
- `qult__complete_wave` — finalize current wave
- `qult__update_task_status` — update task status in tasks.md
- `qult__record_test_pass` — record test pass with command
- `qult__record_review` — record review completion

### SDD Lifecycle
1. `/qult:spec <name> "<description>"` — start spec
2. `/qult:wave-start` → implement → `/qult:wave-complete` — per-wave
3. `/qult:review` — 4-stage independent review at spec end
4. `/qult:finish` — archive and merge/PR/hold/discard

### Workflow Rules
See `.qult/config.json` for project-specific thresholds.
<!-- /@generated -->
```

`npx qult update` はこのマーカーブロックのみを更新し、それ以外の行は保持する。

## Interfaces

### 1. Integration Registry

```typescript
// src/integrations/base.ts
interface IntegrationBase {
  /** ユニーク識別子 (kebab-case) */
  readonly key: IntegrationKey;
  /** 表示名 */
  readonly displayName: string;
  /** プロジェクトルートを受け取り、このツールが使用されているかを判定する */
  detect(projectRoot: string): boolean;
  /** 全設定ファイルを生成する（冪等） */
  generateConfigFiles(ctx: GenerationContext): Promise<void>;
  /** MCP server を各ツールの設定ファイルに登録する */
  registerMcpServer(ctx: GenerationContext): Promise<void>;
}

interface GenerationContext {
  projectRoot: string;
  /** バンドル済みテンプレートルートまたはカスタム dir */
  templateRoot: string;
  /** レンダリング変数 */
  vars: Record<string, string>;
  /** 既存ファイルを上書きするか */
  force: boolean;
  /** 既存ファイルへの追記時に確認プロンプトを表示するか (TTY のみ) */
  interactive: boolean;
}
```

**4 つの具象 Integration クラス:**

| クラス | key | 検出マーカー | 生成ファイル |
|--------|-----|------------|-------------|
| `ClaudeIntegration` | `"claude"` | `.claude/` ディレクトリ or `devDeps["@anthropic-ai/sdk"]` | `.claude/commands/*.md`, `CLAUDE.md`, `.mcp.json` |
| `CodexIntegration` | `"codex"` | `.codex/` ディレクトリ | `.codex/config.toml` (`[mcp_servers]`) |
| `CursorIntegration` | `"cursor"` | `.cursor/` ディレクトリ or `devDeps["@cursor/cursor"]` | `.cursor/rules/*.mdc`, `.cursor/mcp.json` |
| `GeminiIntegration` | `"gemini"` | `.gemini/` ディレクトリ | `.gemini/commands/*.toml`, `.gemini/settings.json`, `GEMINI.md` |

**MCP 設定形式の差異:**

各 Integration の `registerMcpServer()` は以下のツール固有形式に変換する:

```typescript
// claude: .mcp.json (JSON)
{
  "mcpServers": {
    "qult": { "type": "stdio", "command": "npx", "args": ["qult", "mcp"] }
  }
}

// codex: .codex/config.toml (TOML)
[mcp_servers.qult]
command = "npx"
args = ["qult", "mcp"]

// cursor: .cursor/mcp.json (JSON, claude と同形式)
{
  "mcpServers": {
    "qult": { "type": "stdio", "command": "npx", "args": ["qult", "mcp"] }
  }
}

// gemini: .gemini/settings.json (JSON)
{
  "mcpServers": {
    "qult": { "command": "npx", "args": ["qult", "mcp"] }
  }
}
```

### 2. Template Renderer

```typescript
// src/templates/renderer.ts

/**
 * agent-neutral テンプレートを変数置換してレンダリングする。
 * 依存ゼロ・単純文字列置換のみ。
 */
function renderTemplate(template: string, vars: Record<string, string>): string;

/**
 * テンプレート内の未定義変数を検出する。
 * 1 つでもあれば Error を throw する。
 */
function detectUndefinedVars(template: string, vars: Record<string, string>): void;
```

`{{VAR}}` の `VAR` は `[A-Z0-9_]+` で構成される大文字英数字アンダースコア。正規表現 `/\{\{([A-Z0-9_]+)\}\}/g` で全プレースホルダーを抽出し、`vars` に存在しないものがあれば `UndefinedVariableError` を throw する。条件分岐・ループは非対応（KISS）。

**agent-neutral テンプレートから各ツール形式への変換:**

| 変換 | 戦略 |
|------|------|
| commands/ `.md` → claude `.md` | そのままコピー（Claude の commands は markdown） |
| commands/ `.md` → gemini `.toml` | TOML ヘッダー（`[command]` セクション）付きで変換 |
| commands/ `.md` → codex | AGENTS.md テキスト指示として統合（ファイル個別ではない） |
| commands/ `.md` → cursor `.mdc` | frontmatter（`---`）を付与して `.mdc` に変換 |
| rules/ `.md` → claude rules | そのままコピー |
| rules/ `.md` → cursor `.mdc` | frontmatter 付き `.mdc` に変換 |
| rules/ `.md` → gemini `.md` | そのままコピー（GEMINI.md から参照） |
| constitution.md | すべての integration が AGENTS.md に内容を組み込む |

### 3. CLI サブコマンド

```typescript
// src/cli/index.ts — エントリポイント

interface SubcommandHandler {
  command: string;
  description: string;
  options: OptionDef[];
  run(args: ParsedArgs): Promise<void>;
}
```

**5 つのサブコマンド:**

| サブコマンド | 主な処理 |
|------------|---------|
| `init` | 検出/選択 → AGENTS.md 生成 → integration ファイル生成 → config 書き込み |
| `update` | config から enabled 読み込み → `@generated` ブロック更新 |
| `check` | state 読み込み → detector 実行（`--detect`） → 出力 |
| `add-agent` | 単一 integration 追加（`--force` で上書き） |
| `mcp` | MCP server 起動（stdio JSON-RPC ループ） |

引数パーサは依存ゼロの手書き実装（`process.argv` を走査）。オプション数が少ないため `cac` / `yargs` は採用しない。

対話 UI: `@clack/prompts` を **採用しない**。TTY 判定 (`process.stdout.isTTY`) + `node:readline` の `question()` で selectbox を自作する。理由: `@clack/prompts` は `dependencies` に入ることになり「依存ゼロ原則」に反する。

```typescript
// src/cli/prompt.ts — 手書き選択 UI（TTY のみ使用）
async function select<T>(
  message: string,
  options: Array<{ label: string; value: T }>,
): Promise<T>; // non-TTY では呼ばれない
```

### 4. MCP Server

```typescript
// src/mcp/server.ts — stdio JSON-RPC ループ（Node.js版）
```

既存 `src/mcp-server.ts` のロジックをそのまま移植する。`Bun.stdin` / `Bun.stdout` を `process.stdin` / `process.stdout` に、`createInterface` (already `node:readline`) のまま維持する。Tool 一覧は現在と同一:

| カテゴリ | ツール名 |
|---------|---------|
| Spec | `get_active_spec`, `complete_wave`, `update_task_status`, `archive_spec`, `record_spec_evaluator_score` |
| State | `get_project_status`, `record_test_pass`, `record_review`, `record_stage_scores`, `record_human_approval`, `record_finish_started` |
| Detector | `get_pending_fixes`, `clear_pending_fixes`, `get_detector_summary`, `get_file_health_score`, `get_impact_analysis`, `get_call_coverage` |
| Gate/Config | `disable_gate`, `enable_gate`, `set_config` |

起動コマンド: `npx qult mcp` → `node dist/mcp-server.mjs` 相当のエントリポイントを実行。

### 5. Detector

```typescript
// src/detector/index.ts
interface DetectorResult {
  detector: string;
  fixes: PendingFix[];
  skipped: boolean;
  skipReason?: string;
}

async function runAllDetectors(files: string[], opts: DetectorOptions): Promise<DetectorResult[]>;
```

**5 種の Detector と Node.js 移植評価:**

| Detector | 移植難度 | 主な変更点 |
|---------|---------|----------|
| `security-check` | 低 | `Bun.file` → `readFileSync` (既に `node:fs` 使用済み) |
| `dep-vuln-check` | 低 | `execFileSync` は既に `node:child_process` |
| `hallucinated-package-check` | 低 | `fetch` は Node 18+ 標準。Bun 固有 API なし |
| `test-quality-check` | 低 | `readFileSync` 済み。Bun 依存なし |
| `export-check` | 低 | `execSync` / `readFileSync` 済み。Bun 依存なし |

現行実装は既にほぼ `node:*` API のみ使用しており、移植コストは低い（主な変更は `import` パスの調整）。

**ネットワーク不可時のスキップ実装:**

```typescript
async function isNetworkAvailable(): Promise<boolean> {
  try {
    const ctrl = new AbortController();
    setTimeout(() => ctrl.abort(), 2000);
    await fetch("https://registry.npmjs.org/-/ping", { signal: ctrl.signal, method: "HEAD" });
    return true;
  } catch { return false; }
}
```

`dep-vuln-check` と `hallucinated-package-check` は実行前に `isNetworkAvailable()` を呼び、`false` なら `DetectorResult { skipped: true, skipReason: "network unavailable" }` を返す。

## Dependencies

### runtime dependencies（`dependencies`: 空を維持）

外部ランタイム依存はゼロ。Node.js 20+ の標準ライブラリのみ使用:

- `node:fs` / `node:path` / `node:readline` / `node:child_process` / `node:process`
- グローバル `fetch`（Node 18+）

### devDependencies

| パッケージ | 役割 | 既存/新規 |
|-----------|------|---------|
| `typescript` | 型チェック | 既存 |
| `@biomejs/biome` | lint/format | 既存 |
| `vitest` | テスト | 既存（`bunx --bun vitest` → `npx vitest` に変更） |
| `tsup` | ビルド | **新規**（`bun build.ts` を置き換え） |
| `@types/node` | Node.js 型定義 | **新規**（`@types/bun` を置き換え） |

### tsup の採用理由

- 既存 `bun build.ts` は `Bun.build()` API を使用しており Node.js では動作しない
- `tsup` は esbuild ベースで ESM 形式を出力でき、CLI + MCP server の 2 エントリポイントビルドに対応
- `esbuild` 直接使用より `tsup` の方が設定が簡潔（KISS）
- `tsup` は devDependency のみで runtime には含まれない

## Alternatives Considered

- **`commander` / `yargs` での引数パース** — rejected: runtime dependency になる。5 サブコマンド・限定的なオプション数であれば手書きで十分（KISS）

- **`@clack/prompts` での対話 UI** — rejected: runtime dependency になる。`node:readline` の `question()` + 手書き select UI で機能的に代替可能。デザイン品質は若干低下するが、依存ゼロ原則を優先

- **`handlebars` / `mustache` でのテンプレート** — rejected: runtime dependency になる。`{{VAR}}` 単純置換のみで十分（条件分岐・ループは Out of Scope）。関数 1 つで実装可能

- **`toml` パッケージでの TOML 生成** — rejected: runtime dependency になる。`codex` integration の `.codex/config.toml` と `gemini` の `.toml` コマンドファイルは、テンプレートを文字列としてそのまま配置するアプローチで対応（TOML のパース・生成は不要。既存 `settings.json` を JSON で読み書きするだけ）

- **`jest` / `mocha` でのテスト** — rejected: `vitest` を継続使用。既存テストコードの書き換えが不要で、移行コストが最小

- **SQLite / グローバル設定** — rejected: Out of Scope（requirements.md 明示）。ファイルベース state を継続

- **`generic` integration（AGENTS.md のみ）** — rejected: Out of Scope（requirements.md 明示）。全 integration で AGENTS.md は共通生成されるため代替十分

- **手書き esbuild スクリプト（Node.js 版）** — rejected: `tsup` の方が設定が少なく maintainability が高い。`bun build.ts` の置き換えとして `tsup` が最適

## Risks

### R1: Bun 固有 API の残存

**リスク:** `src/` に `Bun.*` 呼び出しが残存し、Node.js ビルド時に実行時エラーになる。

**緩和策:** `tsup` のビルドで `@types/bun` を除去し `@types/node` のみにすることで、`Bun.*` の使用がコンパイルエラーになる。CI で `node dist/cli.mjs --version` の smoke test を実行する。

### R2: `plugin/` 削除による既存ユーザーへの影響

**リスク:** v1.x ユーザーは `~/.claude/rules/qult-*.md` が消えたり MCP server が動かなくなる。

**緩和策:** requirements.md で「後方互換なし」と明示済み（architect 1 名のみが既存ユーザー）。Wave 末尾での `plugin/` 削除は後半 Wave（Wave 6）に配置し、早期の影響を防ぐ。

### R3: 各 AI ツールの MCP 設定ファイル形式の変更

**リスク:** Cursor / Gemini / Codex が MCP 設定フォーマットを変更し、生成ファイルが無効になる。

**緩和策:** MCP 設定スニペットは Integration クラス内の定数として集中管理する。形式変更時は単一ファイルの修正で対応可能。

### R4: TOML 生成の手書き脆弱性

**リスク:** `codex` / `gemini` の `.toml` ファイルをテンプレートとして配置する場合、プロジェクト名等の変数に TOML 特殊文字（`"`, `\n` 等）が含まれると形式が壊れる。

**緩和策:** `{{PROJECT_NAME}}` 等の変数として挿入する値は英数字・ハイフン・アンダースコアのみを許容するバリデーションを加える。TOML では文字列値を必ずダブルクォートで囲むテンプレートとする。

### R5: Node.js fetch のタイムアウト挙動の差異

**リスク:** Bun の `fetch` と Node.js 18/20 の `fetch` でタイムアウト・キャンセル挙動が異なり、ネットワーク確認ロジックがハングする。

**緩和策:** `AbortController` + `setTimeout` パターンは Node.js 20 で動作確認済み。`hallucinated-package-check` の既存実装がこのパターンを使用しており移植リスクは低い（推測: 低リスク）。

### R6: `npx qult mcp` の起動遅延

**リスク:** `npx` の起動オーバーヘッドがあり、MCP server との通信開始が遅延する。Claude Code 等の AI ツールがタイムアウトする可能性。

**緩和策:** MCP 設定ファイルには `command: "npx", args: ["qult", "mcp"]` を基本とし、グローバルインストール時 (`npm i -g qult`) は `command: "qult", args: ["mcp"]` を許容する。`npx qult mcp` は初回 npm キャッシュ後は遅延がほぼなく、実用上問題ない（推測: 低リスク）。
