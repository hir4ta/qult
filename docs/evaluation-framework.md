# qult 自己評価フレームワーク

AI コーディングハーネスとしての qult の成熟度を定期的に評価するためのフレームワーク。各カテゴリは最新の学術研究・産業レポートに基づく。

## 評価カテゴリ

### 1. ポリシー強制（重み: 20%）

エージェントがルールを構造的に遵守することを保証する能力。

**評価項目**:

- プロンプトではなく実行境界（リファレンスモニタ）で強制しているか
- ネゴシエーション不可能な DENY メカニズムが存在するか
- エージェントがルールを「好み」として扱う余地がないか
- ドリフト回復メカニズムがあるか（回復率 > ドリフト率の保証）
- 制約違反の検出率と対応の即時性
- ポリシーの段階的エスカレーション（advisory → blocking）

**参考文献**:

- AgentSpec (arXiv:2602.22302) — Drift Bounds 定理
- PCAS (arXiv:2602.16708) — リファレンスモニタパターン
- AgentPex (arXiv:2603.23806) — 83% のトレースに手続き的違反
- ODCV-Bench (arXiv:2512.20798) — 制約違反率 30-50%

### 2. セキュリティ（重み: 15%）

AI 生成コードのセキュリティ脆弱性を検出・防止する能力。

**評価項目**:

- OWASP Top 10 パターンのカバレッジ
- 反復改善によるセキュリティ劣化への対策（advisory→blocking 昇格）
- SAST ツール統合（Semgrep 等）の有無と強制力
- シークレットスキャン（ハードコードされた API キー、トークン等）
- サプライチェーンセキュリティ（依存関係の脆弱性チェック）
- ソースマップ・デバッグ情報の露出防止
- SAST + LLM のハイブリッドトリアージ精度
- セキュリティ検出の偽陽性率と偽陰性率のバランス

**参考文献**:

- Security Degradation in Iterative AI Code (arXiv:2506.11022) — 反復で脆弱性 37.6% 増
- GitGuardian 2026 — AI コミットのシークレットリーク率 2x
- VibeGuard (arXiv:2604.01052) — 5 つの盲点
- Semgrep + LLM Hybrid — 精度 35.7% → 89.5%

### 3. コード品質検出（重み: 15%）

AI 生成コードの品質問題を検出する能力。

**評価項目**:

- 未使用コード・デッドインポートの検出
- コード重複の検出（ファイル内・ファイル間）
- セマンティックパターン（空の catch、loose equality 等）
- 命名規則ドリフトの検出
- テスト品質の検出（空テスト、自明なアサーション等）
- プロジェクト全体のコードヘルス指標の有無
- 検出器の網羅性（パターン数・カバー範囲）
- 検出器間の相互補完性

**参考文献**:

- CodeScene — AI 向けコードヘルス閾値 9.5+
- CodeRabbit — AI コードは 1.7x の問題
- Triple Debt Model (arXiv:2603.22106) — 認知負債・意図負債
- SonarQube Study (arXiv:2508.14727) — モデル横断的な系統的弱点

### 4. コンテキストドリフト対策（重み: 10%）

長時間セッションでのコンテキスト喪失・指示忘れを防ぐ能力。

**評価項目**:

- compaction 後の状態再注入メカニズム
- DENY メッセージへの状態サマリー付加
- セッション横断のパターン学習（flywheel）
- ハンドオフドキュメント生成
- コンテキスト予算の効率的使用

**参考文献**:

- Context Drift Kills Agents (Zylos) — 障害の 65% がドリフト、35 分で劣化
- Anthropic: Context Engineering — コンテキストエンジニアリング体系
- Agent Drift (arXiv:2601.04170) — semantic/coordination/behavioral drift

### 5. コードレビュー（重み: 15%）

AI によるコードレビューの品質と実効性。

**評価項目**:

- レビュー結果の強制力（無視不可能か）
- 独立した複数ステージのレビュー構成
- 決定論的検出器の結果を AI レビューの ground truth として活用
- レビュアーモデルの多様性（設定可能性）
- claim grounding（エージェント出力の検証）
- レビューコメントの Succinctness/Accuracy/Actionability フィルタリング
- 偽陽性の抑制メカニズム
- セッション横断のレビュースコアトレンド分析

**参考文献**:

- Nonstandard Errors (arXiv:2603.16744) — モデル多様性の根拠
- SWE-PRBench (arXiv:2603.26130) — 人間指摘の 15-31% のみ検出
- Specification as Quality Gate (arXiv:2603.25773) — 決定論的ゲート先行
- LLM Code Review (arXiv:2505.20206) — 偽陽性率 24.80%

### 6. TDD・仕様駆動（重み: 10%）

テスト駆動開発と仕様ベースの品質保証を構造的に強制する能力。

**評価項目**:

- テストファイル先行編集の強制（RED→GREEN→REFACTOR）
- Verify フィールドによるタスク完了時のテスト自動実行
- 仕様ゲーミング防止策（hidden test splits 等）
- プランベースのタスク管理と完了条件の構造化
- テストランナーの自動検出と統合
- テスト品質の定量的評価

**参考文献**:

- TDAD (arXiv:2603.17973) — プロンプト TDD はリグレッション悪化
- TDAD Compilation (arXiv:2603.08806) — hidden test splits で仕様ゲーミング防止
- Simon Willison: Red/Green TDD — 構造的強制パターン

### 7. マルチエージェント協調（重み: 5%）

複数のエージェント（レビュアー、プランナー等）の協調品質。

**評価項目**:

- エージェント間の矛盾検出
- エージェント数の適切性（4 エージェント超で協調オーバーヘッド）
- 独立エージェント vs 合議型の設計判断
- サブエージェント出力の検証メカニズム
- エージェント障害モードのカバレッジ（MAST 14 障害モード）

**参考文献**:

- Multi-Agent Teams Hold Experts Back (arXiv:2602.01011) — 最大 37.6% の性能低下
- DeepMind: Scaling Agent Systems (arXiv:2512.08296) — 4 エージェント超でオーバーヘッド
- MAST (arXiv:2503.13657) — 14 障害モード分類

### 8. アーキテクチャ設計（重み: 10%）

ハーネスとしての構造的健全性。

**評価項目**:

- fail-open の一貫性（ハーネスの障害で開発を止めないか）
- ガイド（フィードフォワード）+ センサー（フィードバック）の分離
- 決定論的（computational）と推論的（inferential）の使い分け
- 状態管理の原子性と一貫性
- 外部依存の最小化
- ビルド・配布の簡潔さ
- テストカバレッジとシミュレーションテスト
- ランタイムオーバーヘッドの可視化と制御

**参考文献**:

- Martin Fowler: Harness Engineering — ガイド/センサー分類体系
- Anthropic: Managed Agents — 頭脳と実行の分離
- From Governance to Enforceable Controls (arXiv:2604.05229) — 4 制御層

## 横断的チェック項目

以下はカテゴリ横断で確認すべき項目。

### ガバナンス層の網羅性（LGA 参照）

| 層 | 内容 | チェック観点 |
|----|------|------------|
| L1 実行サンドボックス | ツール実行の隔離 | gate コマンドの安全な実行、shell injection 防止 |
| L2 意図検証 | アクションの妥当性検証 | pre-tool の DENY 判定ロジック |
| L3 エージェント間認可 | サブエージェントの信頼境界 | claim grounding、出力検証 |
| L4 監査ログ | 不変の操作記録 | audit-log の網羅性と改ざん耐性 |

### ナレッジアーキテクチャ（Knowledge Activation 参照）

- skills がドメイン知識の再利用可能な単位として機能しているか
- 新規ユーザーのオンボーディングパスが明確か
- ワークフローの選択肢が多すぎて認知負荷を増やしていないか

### 生産性への影響

- ハーネスが開発速度に与えるオーバーヘッドを計測しているか
- false positive による開発者の疲労（alert fatigue）を管理しているか
- ゲートのスキップ（escape hatch）が適切に提供されているか

## 評価の実施方法

1. 各カテゴリの評価項目を 0-10 で採点
2. 重みを掛けて加重平均を算出
3. 前回評価との差分を記録
4. 弱み（7 未満）のカテゴリに対して改善計画を策定
5. 新しい参考文献が出たら評価項目を更新
